# Research Policy

## Purpose

Research in this repo exists to reduce architectural guesswork and leave durable evidence behind for future Codex sessions.

## Required note structure

Every note should include:

1. `## Date`
2. `## Summary`
3. `## Primary sources`
4. `## Why this matters to this repo`
5. `## Decision impact`

## Source policy

- Official docs, engine vendor guidance, standards docs, and first-party samples come first.
- Secondary commentary is optional support, not the main decision basis.
- If a note uses unstable or time-sensitive guidance, include the exact date and version family.

## Repo integration

- Link the note from the related active doc or task workflow.
- Turn stable architectural lessons into starter-kit rules, checklist items, or validation commands.
- Treat research without repo impact as incomplete.
# Game Development Research Policy

This folder is the durable, source-backed reasoning layer for the studio system.

## Source policy

- Prefer official engine documentation, platform-holder guidance, standards bodies, or first-party developer documentation first.
- When a note makes broader design, UX, AI, motivation, accessibility, or tuning claims, add at least one canonical academic or standards reference when available.
- If a note references a commercial game as an example, treat it as a design contrast set, not as proof.
- Avoid unsupported claims like "this is best practice" without a source or a repo-specific validation story.

## Required note shape

Every durable note should include:

- `## Date`
- `## Summary`
- `## Primary sources`
- `## Why this matters to this repo`
- `## Decision impact`

## Repo expectation

- Research should change something durable: a checklist, a starter-kit rule, a feature brief, an active doc, a validation story, or a routing heuristic.
- If a note does not change the repo's behavior or decisions, it is probably not worth maintaining.
