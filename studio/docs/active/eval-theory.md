# Eval Plan: Theory Stack

## Change under test

- Theory guide and example
- Theory research note
- Theory checklist
- Theory routing and checklist resolution

## Goal

- Keep theory tasks anchored in explicit lens stacks instead of vague design language.
- Make sure the agent can reopen the same theoretical model later.

## Risks

- The lane can become a dumping ground for generic design talk if the lens stack is not enforced.
- Theory can drift into implementation details if the boundary is not explicit.
- Strong-sounding language can hide weak evidence unless the validation path stays narrow.

## Validation

- Route a theory-shaped task and confirm the theory lane opens.
- Resolve the theory checklist bundle and confirm the required items are present.
- Check that guide, example, research note, checklist, and eval plan all point to each other.

## Success criteria

- Player outcome and dominant tension are explicit.
- Lens stack is named before implementation.
- Evidence is visible.
- One narrow proof path exists.
